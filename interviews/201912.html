指针与引用

  • 指针, 首先, 是实体，它指向一块内存，它的内容就是所指的 内存的地址；而引用的话是某块内存的别名
  • 然后 引用只能在定义时被初始化一次，之后不可以改变的；而指针是可以改变的
  • 另外还有个区别是 const, 引用没有 const，指针有 const，const 的指针不可变；
  • 引用不能为空，指针可以为空；
  • “sizeof 引用”得到的是所指向的变量（对象）的大小，而“sizeof 指针”得到的是指针本身的大小，（32位下 即4个字节）；
  • 指针和引用的自增(++)运算意义不一样；
  • 指针可以有多重指针，引用没有。

linux 分配内存的方式 (malloc, calloc, realloc, _alloca)

  • malloc: 分配长度为num_bytes字节的内存块
  • calloc: 跟malloc的区别：calloc在动态分配完内存后，自动初始化该内存空间为零，而malloc不初始化，里边数据是随机的垃圾数据。
  • realloc: realloc是从堆上分配内存的，当扩大一块内存空间时，realloc会直接从堆上现存的数据后面的那些字节中获得附加的字节，先判断当前的指针是否有足够的连续空间，如果有，扩大mem_address指向的地址，并且返回原地址。
  • _alloca: 内存分配函数,与malloc, calloc, realloc类似. 但是注意一个重要的区别,_alloca是在栈(stack)上申请空间,用完马上就释放.

malloc 和 free 的区别

malloc与free是C++/C语言的标准库函数，new/delete是C++的运算符。它们都可用于申请动态内存和释放内存。
此外,new是强制类型的,malloc不是,需要类型转换
当然还有很多不同
new 可以调用构造函数在声明的时候初始化
malloc只是分配空间,需要在其他地方初始化
而delete不仅会释放空间,在释放前会调用析构函数
而且malloc需要指定分配空间大小, 而new是自动计算的

TCP 和 UDP 的区别

1）连接方面区别
TCP面向连接（如打电话要先拨号建立连接）。
UDP是无连接的，即发送数据之前不需要建立连接。
2）安全方面的区别
TCP提供可靠的服务，通过TCP连接传送的数据，无差错，不丢失，不重复，且按序到达。
UDP尽最大努力交付，即不保证可靠交付。
3）传输效率的区别
TCP传输效率相对较低。
UDP传输效率高，适用于对高速传输和实时性有较高的通信或广播通信。
4）连接对象数量的区别
TCP连接只能是点到点、一对一的。
UDP支持一对一，一对多，多对一和多对多的交互通信。

函数覆盖（重载）和虚函数的区别

虚函数可以在程序运行的时候实现多态（调用虚函数时通过调用虚表来确定最终调用哪个实际函数）可以说是“动态”的多态；而函数重载（即你说的函数覆盖）是在源代码编译的时候，编译器就可以明确调用哪个函数的，实际上是属于“静态”的多态。

什么变量放在栈上，什么放在堆上

局部的变量 函数的形参 会存放在栈中。
全局变量 静态的局部变量 还有静态的全局变量 会存放在堆中。

堆(heap)和栈(stack) 有什么不同

  • 没有特殊情况，用栈分配。
  • 如果想让资源的生命周期超过对应变量的作用域，用堆分配
  • 如果资源太大，或者只有运行时才知道大小，用堆分配
  • 我觉得zhan区(stack)是由编译器自动去分配, 自动去释放的 ,存放方法(函数)的参数值, 局部变量的值等，栈是向低地址扩展的数据结构，所以可以说 它是一块连续的内存区域。所以栈顶部的地址和栈的最大容量都是系统它预先规定好的。
    堆区(heap)一般由程序员分配释放, 若程序员不释放,就是说它整个程序结束时由OS回收，向高地址扩展的数据结构，是不连续的内存区域，从而可以说堆获得的空间比较灵活。

MVC:

它被分为三个核心部件：模型、视图、控制器。下面是每一个部件的分工：视图是用户看得到的 并且用户能与之交互的界面。模型 表示的是 业务数据，提供了数据 给 视图。控制器接受用户的输入并调用模型和视图去完成用户的需求。

然后 MVC 的 控制流程 是这样的：用户的请求是发送到控制器的。然后控制器依赖请求去选择加载哪个模型，并把模型附加到对应的视图。附加了模型数据的最终视图做为响应发送给终端用户。

MVC有两个大的好处：

  1. 我觉得分离了关注点吧。后台代码可以被移到单独的类文件，我们可以最大限度的重复利用代码。
  2. 我会说自动化 UI 测试更加方便，因为后台代码移到了.NET类。这让我们更容易做单元测试和自动化测试。
  3. 相比Windows应用，MVC 架构更适用于 Web 应用。对于Windows应用，MVP(Model View Presenter)架构更好一点。如果你使用WPF和Silverlight，MVVM更适合。

Hadoop 压力测试 (MapReduce)

Hadoop 自带了几个基准的测试

  1. 比如说, 测试 写 的速度。向 (HDFS) 文件系统中写入数据，10 个文件，每个文件大小固定
  2. 另外还有 就是 测试 读文件 的速度: 从 (HDFS) 文件系统中读入 10 个文件，每个文件大小固定
  3. 还有 在 HDFS上 创建、读取、重命名 或者进行 删除文件的操作: 这相当于是测试 NameNode 的负载，它会生成很多与HDFS相关的请求，给NameNode施加较大的压力。 (NameNode: 管理文件系统的命名空间。它维护着文件系统树及整棵树内所有的文件和目录)
  4. 当然还有 MapReduce 多次重复执行一个小作业 比如说 单词计数，这主要用来检查在机群上小作业的运行是否可重复以及运行是否高效。

Wordcount:

  1. mapper: 将单词输出为一个个 pair 的格式, 比如说 you 这个词在 how are you 里面出现了一次, 就输出成 {you, 1}
  2. reducer阶段是汇总各个 key 的个数 并且输出该 key 的总数
  3. 最后 driver 阶段需要 关联mapper和reducer类 输出各个单词出现的总数是多少

HDFS 和 MapReduce 优缺点 (HDFS负责分布式存储，而MapReduce负责分布式计算):

  1. HDFS 的优势
    1. 处理超大文件
    2. 运行于廉价的商用机器上
    3. 容错性比较高 一份数据会自动保存多个副本
    4. 一个数据集一旦有数据源生成，就会被复制分发到不同的存储节点中 所以访问数据会更搞笑
  2. HDFS 的局限
    1. HDFS 主要是为了 达到高的数据吞吐量 而设计的, 对低延迟的访问不太友好
    2. 同理, HDFS 不太适合存储大量的小文件
    3. 多用户写入和随机文件的修改比较慢
  3. MapReduce 的优势
    1. 对程序员比较友好 易于理解 只需要关心应用层怎么写 因为 mapreduce 通过 抽象模型和计算框架 给开发和提供了接口
    2. 扩展性比较好 只需要通过简单的增加机器来扩展它的计算能力
    3. 高容错性 (比如，其中一台机器宕机了，它可以把上面的计算任务转移到另一个节点上运行，不至于这个任务运行失败)
  4. MapReduce 的局限
    1. 实时计算不太擅长 无法像 MySQL 那样在毫米内返回结果
    2. 动态化输入数据不行, mapreduce 输入数据时是静态的
    3. 有向图计算比较慢 因为 多个应用程序 存在依赖关系 导致每个MapReduce作业的输出结果都会写入磁盘 导致性能非常低下

Cloud Foundry

  1. 介绍: Cloud Foundry是一个开源的 平台即服务(PaaS)，它提供给了开发者很大的自由度去选择云平台、开发框架和应用服务。我想先说一下它的起源吧, Cloud Foundry最早由VMware发起的，虽说现在已不在. 但它在当时 业界 得到了广泛的支持 和 传播，它能够让开发者更快更容易的去 开发、测试、部署和扩展应用。Cloud Foundry让开发人员专注于编写应用程序，而不是去为 [中间件] 和 [基础设施] 分心。它提供了 自助式的 高生产力的 框架和应用服务，开发人员在这个基础上可以快速在自己的环境上开发和测试自己的下一代应用，并能部署到云上 而不用去做任何更改。
  2. why CF: 帮助屏蔽了管理中间件和运行部分，让我们集中在应用和数据。底层还有IaaS云，帮你屏蔽了管理操作系统。
  3. 特点: 支持公共，私有和混合云上部署; 支持业界标准框架(Spring for Java); 支持应用服务(mongoDB); 架构可扩展; 开源项目社区
  4. 核心组件:
    1. Router(对所有进来的请求进行路由): Router是整个平台的流量入口，负责分发所有的请求到对应的组件，包括来自外部用户对app的请求和平台内部的管理请求。
    2. Cloud Controller(管理模块): Cloud Controller负责管理app的整个生命周期。用户通过命令行工具cf与CloudFoundry Server打交道，实际主要就是和Cloud Controller交互。
    3. Services(插件): app在运行的时候通常需要依赖外部的一些服务，比如数据库服务、缓存服务、短信邮件服务等等。Service Broker就是app接入服务的一种方式
      1. 在本地开发云应用程序

      3. 节省复制粘贴云应用程序 URL 通常所需的时间

MVC

它被分为三个核心部件：模型、视图、控制器。下面是每一个部件的分工：

视图是用户看得到的 并且用户能与之交互的界面。
模型 表示的是 业务数据，提供了数据 给 视图。
控制器接受用户的输入并调用模型和视图去完成用户的需求。

然后 MVC 的 控制流程 是这样的：

用户的请求是发送到控制器的。然后控制器依赖请求去选择加载哪个模型，并把模型附加到对应的视图。附加了模型数据的最终视图做为响应发送给终端用户。

MVC有两个大的好处：

分离了关注点。后台代码被移到单独的类文件，我们可以最大限度的重复利用代码。
自动化UI测试成为可能，因为后台代码移到了.NET类。这让我们更容易做单元测试和自动化测试。

相比Windows应用，MVC架构更适用于Web应用。对于Windows应用，MVP(Model View Presenter)架构更好一点。如果你使用WPF和Silverlight，MVVM更适合。

Human Gesture Recognition 💊

Definition

通过 佩戴一个肌肉探测手环, 结合 Data mining, ML 去判断一个人用勺吃饭和用叉子吃饭的区别. 手腕装置叫 Myo Armband, 能够根据你的肌肉动作 返回十几种不同的参数. 首先我们先取样, 拍摄多人分别用勺子和叉子吃饭的视频, 然后我们根据视频的动作来截取不同的时间线, 比如多少多少秒开始用叉子 多少多少秒开始用勺子 根据时间线 timestamp 来整理和分类手腕上的数据 数据先是通过 Feature Extraction 特征检测, 比如 平均数, 方差, 中间值等 进行比较, 再根据 PCA 来降维. 降维可以使数据集更易使用并且能有去噪的效果, 也使得结果更容易理解. 把 PCA 得到的结果分为两个部分, 60%作 train 40%作测试. 最后用了四种不同的 ML 方法来训练降维后的数据.(SVM, DT, Naïve Bayes, Neural networks for pattern recognition), 精确率（Precision）、召回率（Recall）、F1-score、ROC、AUC, 得出的对比结论返回到手机设备上.

PCA

PCA(principal Component Analysis)，是一种使用比较广泛的 数据压缩算法。在PCA中，数据从原来的坐标系转换 成了 新的坐标系，这个新的坐标系是由数据本身决定。转换坐标系时，以方差最大的方向作为坐标轴方向，因为数据的最大方差给出了数据的最重要的信息。然后 第一个新坐标轴 选择的是 原始数据中 方差最大的 那个方法，第二个新坐标轴选择的是与第一个新坐标轴方差第二大的方向。最后就一直重复该过程，重复次数为原始数据的(特征)维数。

PCA 降维

  1. 降维是对数据 高维度特征 的一种预处理方法。
  2. 降维是将 高维度的数据 保留下 最重要的一些特征，比如去除噪声和一些不重要的特征，从而实现提升数据处理速度的目的。
  3. 然后其实在实际的生产和应用中啊，降维在一定的信息损失范围内，可以为我们节省大量的时间和成本。也是一种比较广泛的数据预处理方法。
    降维具有如下一些优点：(1）使得数据集更易使用（2）降低算法的计算开销（3）去除噪声（4）使得结果容易理解

SVM

  1. 在特征空间上找到最佳的分离超平面使得训练集上正负样本间隔最大; 它在解决小样本、非线性及高维模式识别中表现出许多特有的优势
  2. 首先通过非线性变换将输入空间变换到一个高维的空间，然后在这个新的空间求最优分类面即最大间隔分类面，而这种非线性变换是通过定义适当的内积核函数来实现的。SVM实际上是根据统计学习理论依照结构风险最小化的原则提出的，要求实现两个目的：
    1. 两类问题能够分开（经验风险最小）
    2. margin最大化（风险上界最小）既是在保证风险最小的子集中选择经验风险最小的函数。

DT

决策树是一种基本的分类与回归方法。其主要有点事模型具有可得性，分类速度快。学习时，利用训练数据，根据损失函数最小化原则建立决策树模型；预测时，对新数据，利用决策树模型进行分类。

  1. 选择特征
  2. 决策树生成
  3. 剪枝
    顾名思义，决策树model是树形结构，在分类中，表示基于特征对实例进行分类的过程。可以认为是“if-else”的合集，也可以认为是特征空间，类空间上条件概率分布。主要优点是分类速度快，可读性好。在学习时（training）根据loss function最小化原则建立决策树model，预测时对新数据利用决策树进行分类。常包括三个步骤*：特征选择，决策树生成，决策树剪枝

Naïve Bayes

在机器学习中，朴素贝叶斯分类器是一个基于贝叶斯定理的比较简单的概率分类器，其中 naive（朴素）是指的对于模型中各个 feature（特征） 有强独立性的假设，并未将 feature 间的相关性纳入考虑中。

朴素贝叶斯分类器一个比较著名的应用是用于对垃圾邮件分类，通常用文字特征来识别垃圾邮件，是文本分类中比较常用的一种方法。朴素贝叶斯分类通过选择 token（通常是邮件中的单词）来得到垃圾邮件和非垃圾邮件间的关联，再通过贝叶斯定理来计算概率从而对邮件进行分类。

Docker

Docker 应用打包工具

Docker的话, 打个比方, Java号称 “一次编译，到处运行”，是因为java虚拟机解决了平台的兼容性问题，所以有java虚拟机的地方就能跑java代码；然而Docker是：“一次封装，到处运行”，因为docker决绝了应用环境的问题，安装了docker的平台就能跑“docker包”，这样就决绝了“开发环境能跑，一上线就崩”的尴尬。

Docker 是一个开源的应用容器引擎，让开发者可以打包他们的应用以及依赖包到一个可移植的容器中，然后发布到任何流行的Linux机器上，也可以实现虚拟化，容器是完全使用沙箱机制，相互之间不会有任何接口。
一个完整的Docker有以下几个部分组成：DockerClient客户端; Docker Daemon守护进程; Docker Image镜像; DockerContainer容器

Docker和传统虚拟化技术的对比

相比传统虚拟机技术，Docker资源占用少，启动更快，很大的方便了项目的部署和运维。
Docker是在操作系统层面上实现虚拟化，复用本地主机的操作系统，传统方式是在硬件的基础上，虚拟出多个操作系统，然后在系统上部署相关的应用。

Docker的优点

  1. 持续部署与测试 (能保持跨环境的一致性, 可以在开发到产品发布的整个过程中使用相同的容器来确保没有任何差异或者人工干预)
  2. 环境标准化和版本控制 (Docker容器还可以像git仓库一样，可以让你提交变更到Docker镜像中并通过不同的版本来管理它们)
  3. 多云平台 可移植性 (如果运行在亚马逊EC2实例上的Docker容器能够很容易地移植到其他几个平台上，比如说VirtualBox)
  4. 隔离性 (假如在虚拟机上运行了很多程序分散在不同的端口上, 有时候可能会用Apache来做反向代理. 随着环境的改变, 会需要一个内容管理系统. 问题来了, 这时候软件可能需要一个不同版本的Apache Tomcat, 为了满足这个需求，你只能将你现有的软件迁移到另一个版本的Tomcat上，或者找到适合你现有Tomcat的内容管理系统（Alfresco）版本。Docker能够确保每个容器都拥有自己的资源，并且和其他容器是隔离的)
  5. 安全性 (Docker确保运行在容器中的应用程序和其他容器中的应用程序是完全分隔与隔离的，在通信流量和管理上赋予你完全的控制权。Docker容器不能窥视运行在其他容器中的进程。从体系结构角度来看，每个容器只使用着自己的资源（从进程到网络堆栈）)

React

React 的工作原理

React 会创建一个虚拟 DOM(virtual DOM)。当一个组件中的状态改变时，React 首先会通过 "diffing" 算法来标记虚拟 DOM 中的改变，第二步是调节(reconciliation)，会用 diff 的结果来更新 DOM。

React 有何优点

  1. 只需查看 render 函数就会很容易知道一个组件是如何被渲染的
  2. JSX 的引入，使得组件的代码的可读性更加高，也更容易看懂组件的布局，或者组件之间是如何互相引用的
  3. 额 第三个是 React 支持服务端的渲染，这可以改进 网页端 的 SEO 和 一定性能的提升
  4. 第四点是 React 易于测试 因为 比起模板，渲染函数更加方便的 可以进行 调试和测试
  5. 最后一点我想说 React 只关注 View 层，所以可以和其它任何框架 (如Backbone.js, Angular.js) 一起使用

React 中 Element 与 Component 的区别

简单而言，React Element 是描述屏幕上所见内容的数据结构，是对于 UI 的对象表述。典型的 React Element 就是利用 JSX 构建的声明式代码片然后被转化为createElement的调用组合。而 React Component 则是可以接收参数输入并且返回某个 React Element 的函数或者类。

展示组件(Presentational component)和容器组件(Container component)

  1. 展示组件关心组件看起来是什么。展示专门通过 props 接受数据和回调，并且几乎不会有自身的状态，但当展示组件拥有自身的状态时，通常也只关心 UI 状态而不是数据的状态。
  2. 容器组件则更关心组件是如何运作的。容器组件会为展示组件或者其它容器组件提供数据和行为(behavior)，它们会调用 Flux actions，并将其作为回调提供给展示组件。容器组件经常是有状态的，因为它们是(其它组件的)数据源。

类组件(Class component)和函数式组件(Functional component)

在组件需要包含内部状态或者使用到生命周期函数的时候使用 Class Component ，否则使用函数式组件。

(组件的)状态(state)和属性(props)

  1. State 是一种数据结构，用于组件挂载时所需数据的默认值。State 可能会随着时间的推移而发生突变，但多数时候是作为用户事件行为的结果。
  2. Props(properties 的简写)则是组件的配置。props 由父组件传递给子组件，并且就子组件而言，props 是不可变的(immutable)。组件不能改变自身的 props，但是可以把其子组件的 props 放在一起(统一管理)。Props 也不仅仅是数据--回调函数也可以通过 props 传递。

React 中 keys 的作用

Keys 是 React 用于追踪哪些列表中元素被修改、被添加或者被移除的辅助标识。

React 中的事件处理逻辑

为了解决跨浏览器兼容性问题，React 会将浏览器原生事件（Browser Native Event）封装为合成事件（SyntheticEvent）传入设置的事件处理器中。这里的合成事件提供了与原生事件相同的接口，不过它们屏蔽了底层浏览器的细节差异，保证了行为的一致性。另外有意思的是，React 并没有直接将事件附着到子元素上，而是以单一事件监听器的方式将所有的事件发送到顶层进行处理。这样 React 在更新 DOM 的时候就不需要考虑如何去处理附着在 DOM 上的事件监听器，最终达到优化性能的目的。

react生命周期函数

  2. 运行中状态: componentWillReceiveProps:组件将要接收到属性的时候调用
  3. 销毁阶段: componentWillUnmount:组件即将销毁

简述flux 思想

Flux 的最大特点，就是数据的"单向流动"。1.用户访问 View2.View 发出用户的 Action3.Dispatcher 收到 Action，要求 Store 进行相应的更新4.Store 更新后，发出一个"change"事件5.View 收到"change"事件后，更新页面

何为 redux

Redux 的基本思想是整个应用的 state 保持在一个单一的 store 中。store 就是一个简单的 javascript 对象，而改变应用 state 的唯一方式是在应用中触发 actions，然后为这些 actions 编写 reducers 来修改 state。整个 state 转化是在 reducers 中完成，并且不应该有任何副作用。

redux中间件

中间件提供第三方插件的模式，自定义拦截 action -> reducer 的过程。变为 action -> middlewares -> reducer 。这种机制可以让我们改变数据流，实现如异步 action ，action 过滤，日志输出，异常报告等功能。
常见的中间件：redux-logger：提供日志输出; redux-thunk：处理异步操作; redux-promise：处理异步操作 actionCreator的返回值是promise

redux有什么缺点

1.一个组件所需要的数据，必须由父组件传过来，而不能像flux中直接从store取。
2.当一个组件相关数据更新时，即使父组件不需要用到这个组件，父组件还是会重新render，可能会有效率影响，或者需要写复杂的shouldComponentUpdate进行判断。

常用的HTTP方法

GET： 用于请求访问已经被URI（统一资源标识符）识别的资源，可以通过URL传参给服务器
POST：用于传输信息给服务器，主要功能与GET方法类似，但一般推荐使用POST方式。

OPTIONS：查询相应URI支持的HTTP方法。

206：客户端只是请求资源的一部分，服务器只对请求的部分资源执行GET方法，相应报文中通过Content-Range指定范围的资源。

OSI七层模型详解

通过七个层次化的结构模型使不同的系统不同的网络之间实现可靠的通讯

  1. 物理层: 网卡，网线，集线器，中继器，调制解调器; 为数据链路层提供物理连接
  2. 数据链路层: 网桥，交换机: 负责建立和管理节点间的链路
  3. 网络层: 路由器; 数据链路层的数据在这一层被转换为数据包，然后通过路径选择、分段组合、顺序、进/出路由等控制，将信息从一个网络设备传送到另一个网络设备
  4. 传输层: 因此该层是通信子网和资源子网的接口和桥梁，起到承上启下的作用。(TCP/IP中的TCP协议)
  5. 会话层: ：向两个实体的表示层提供建立和使用连接的方法。将不同实体之间的表示层的连接称为会话。因此会话层的任务就是组织和协调两个会话进程之间的通信，并对数据交换进行管理。
  6. 表示层: 处理用户信息的表示问题，如编码、数据格式转换和加密解密; 提高网络的安全性
  7. 应用层: 文件服务、目录服务、文件传输服务（FTP）、远程登录服务（Telnet）、电子邮件服务（E-mail）、打印服务、安全服务、网络管理服务、数据库服务等; 是计算机用户，以及各种应用程序和网络之间的接口，其功能是直接向用户提供服务，完成用户希望在网络上完成的各种工作。

下面4层（物理层、数据链路层、网络层和传输层）主要提供数据传输和交换功能，即以节点到节点之间的通信为主；第4层作为上下两部分的桥梁，是整个网络体系结构中最关键的部分；而上3层（会话层、表示层和应用层）则以提供用户与应用程序之间的信息和数据处理功能为主。简言之，下4层主要完成通信子网的功能，上3层主要完成资源子网的功能。

  1. IP（Internet Protocol）又称为互联网协议，是为不同计算机网络互联进行通信而设计的协议。运行在网络层，主要是为了将传输不同数据包的不同网络连接起来。

C语言经过编译之后将内存分为以下几个区域：栈, 堆, 全局（静态）存储区, 文字常量区, 程序代码区

java中什么是类，什么是接口

Java程序的基本单位是类，类是对象的抽象，对象是类的实例，或者说对象是类定义的的数据类型的变量。你建立类之后，就可用它来建立许多你需要的对象。Java中的接口是一系列方法的声明，是一些方法特征的集合，一个接口只有方法的特征没有方法的实现，因此这些方法可以在不同的地方被不同的类实现，而这些实现可以具有不同的行为（功能）。
接口的两种含义：一，Java接口，Java语言中存在的结构，有特定的语法和结构；二，一个类所具有的方法的特征集合，是一种逻辑上的抽象。前者叫做“Java接口”，后者叫做“接口”。

java中可不可以多重继承？

答：java 不支持多重继承,要想实现多重继承可以使用接口，接口能变相的实现多重继承。

java中的static是怎么回事？

答：static是静态，如果加载方法前就是一个静态方法。可以直接加类名+方法名引用。

java中垃圾回收是怎么回事？

答：java编程语言不为程序员提供操作符或库函数来释放对象的，因为java运行环境会自动收回那些没有引用与之相连的对象所占的内存。这种内存回收的过程就叫 垃圾回收。垃圾回收器自动定期执行。任何时候你都可以手动通过 system.gc() 调用垃圾回收器。
